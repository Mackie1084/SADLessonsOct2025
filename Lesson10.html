10.1 Introduction

Systems Implementation is the fourth of five phases in the systems development life cycle (SDLC)

Includes application development, testing, documentation, training, data conversion, system changeover, and post-implementation evaluation of the results

10.2 Overview of Application Development

Objective is to translate the logical design into program and code modules that will function properly

The tasks involved in system design produced an overall design and a plan for physical implementation

10.4.1 Top-down Approach

Breaks a system down into its smallest components

Each component is developed separately and then integrated into the overall system

Emphasizes dividing a system into smaller subsystems

Each subsystem is designed, coded, and tested independently

10.4.2 Partitioning

Process of dividing a system into smaller parts

Each part is called a module

Modules can be developed, tested, and maintained independently

Facilitates teamwork and parallel development

Makes it easier to manage complexity

10.4.3 Modular Design

Each module performs a specific function

Modules should be designed to be as independent as possible

Facilitates testing, debugging, and maintenance

Promotes code reuse

10.4.4a Structure Charts

Graphical representation of a system's modules

Shows the hierarchy and relationships between modules

Helps visualize the system's structure

Aids in understanding module interactions

10.4.5 Cohesion and Coupling

Cohesion refers to how closely related and focused the responsibilities of a single module are

High cohesion is desirable because it indicates that a module is well-defined and has a clear purpose

Coupling refers to the degree of interdependence between modules

Low coupling is desirable because it indicates that modules can function independently of one another

10.5 Object-Oriented Application Development

Based on the concept of "objects", which can contain data and code

Emphasizes encapsulation, inheritance, and polymorphism

Programmer makes necessary revisions and updates to class diagrams, sequence diagrams, state transition diagrams, and activity diagrams

Main objective is to translate object methods into program code modules and determine what vent or message will trigger the execution of each module

Types of Testing

10.7.1 Unit Testing

  • Test data

  • Programmers must test programs that interact with other programs and files individually

  • Regardless of who creates the test plan, the project manager or a designated analyst also reviews the final test results

10.7.2 Integration Testing

  • Testing the programs independently does not guarantee that the data passed between them is correct

  • A testing sequence should not move to the integration stage unless it has performed properly in all unit tests

10.7.3 System Testing

  • You should regard thorough testing as a cost-effective means of providing a quality product

  • If conflicting views exist, management will decide whether or not to install the system after a full discussion of the options
